User Manual

Comprehensive Guide to Orthrus DAST

1. Introduction to Orthrus DAST

Orthrus DAST is an advanced Dynamic Application Security Testing (DAST) engine specifically engineered for modern Web APIs. Unlike static code analyzers, Orthrus interacts with your running application by simulating real-world attacker behaviors to discover vulnerabilities such as injection flaws, broken access control, and missing security headers.

The architecture is split into two phases:

  1. Discovery Phase: The engine maps your API using specific "Discoverer" modules to identify all available endpoints, HTTP methods, parameters, and authentication requirements.
  2. Scanning Phase: A suite of specialized security scanners is launched against the discovered attack surface.

2. Discovery Modules

Before testing, Orthrus must understand what to test. Select the module that best fits your API:

Discoverer ID Target URL Format Description & Use Case
openapi https://api.example.com/v3/openapi.json Use Case: You have a Swagger/OpenAPI definition.
Parses the JSON/YAML specification to perfectly map endpoints, required parameters, payloads, and authentication schemes. This is the most accurate discovery mode.
graphql https://api.example.com/graphql Use Case: You have a GraphQL endpoint.
Utilizes the GraphQL introspection query to dump the entire schema, extracting all queries, mutations, object types, and input fields.
well-known https://api.example.com Use Case: You want to find hidden files.
Scans the base URL for predictable paths like /.well-known/, /swagger-ui.html, /.git/, and administrative panels to uncover forgotten resources.
curl curl -X GET "https://api..." Use Case: You want to test a single specific request.
Paste a raw cURL command copied from your browser's Developer Tools. Orthrus will parse the URL, method, headers, and body to test that exact endpoint.
blackbox https://api.example.com Use Case: You have zero documentation.
Performs aggressive directory brute-forcing and parameter fuzzing to blindly map an undocumented API surface.

3. Launching a New Scan

Navigate to the New Scan page from the sidebar to start a security assessment.

  • Target URL: Provide the base URL or specification link depending on your chosen Discoverer (see table above).
  • Discovery Mode: Select the method Orthrus will use to map the API.
  • Include Passed Tests: Check this box if you want the final report to include details about the security tests that were executed but did not result in a vulnerability. Highly recommended for compliance auditing.
Advanced Options

The "Advanced Options" toggle reveals a set of powerful configuration settings for complex testing scenarios:

  • Authentication - Option A (Manual Tokens): Provide a `Primary Bearer Token` to inject into all standard requests. Optionally, provide a `Secondary Bearer Token` belonging to a different user profile: Orthrus will automatically perform Cross-User Broken Object Level Authorization (BOLA) tests by trying to access User A's resources using User B's token.
  • Authentication - Option B (Dynamic OAuth2): Instead of manually pasting tokens, you can configure the OAuth2 Token endpoint (`OAuth2 Token URL`), `Client ID`, `Client Secret`, and `Grant Type`. By providing a comma-separated list of `username:password` credentials, Orthrus will dynamically fetch tokens before starting the scan. Note: If Option B is configured successfully, it overrides the manual tokens from Option A.
  • Scanner Selection: You can explicitly check or uncheck which specific security scanners should run (e.g., skip XSS and Rate Limiting tests but keep SQLi). By default, all available scanners are checked and will be executed.
Warning: By default, all security scanners are executed (SQLi, XSS, Rate Limiting, Missing Headers, GraphQL Injection, SSL/TLS checks, etc.). This can generate a significant amount of traffic. Do not scan production environments without authorization.

4. Results and Reporting

When a scan completes, the results are displayed in a comprehensive dashboard and saved to the in-memory database.

The Grading System

Your API is automatically assigned a security grade based on the highest severity vulnerability found:

  • Grade F: At least one CRITICAL vulnerability (e.g., SQLi, SSTI, Command Injection).
  • Grade D: At least one HIGH vulnerability (e.g., Subdomain Takeover).
  • Grade C: At least one MEDIUM vulnerability (e.g., Missing Rate Limiting).
  • Grade B: At least one LOW vulnerability (e.g., Missing Security Headers).
  • Grade A: No vulnerabilities found!
PDF Export

From the Scan History page, you can export any scan result as a perfectly formatted PDF document.

Confidentiality Notice: The generated PDF report contains sensitive information about your API vulnerabilities, including exact reproduction steps and endpoints. Its distribution is strictly restricted to authorized security personnel.
